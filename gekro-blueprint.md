## PART 1: VISION & POSITIONING

**Site identity:** gekro.com is the personal lab and public voice of an AI engineer who builds real things — trade bots, self-hosted systems, EV experiments, automation pipelines — and documents the journey with depth, honesty, and craft.

**Tone:** Confident but not arrogant. Technical but human. Forward-thinking but grounded in nature and sustainability. This is a senior engineer's digital home, not a marketing site.

**Audience tiers:**
- **Primary:** Developers, AI practitioners, builders discovering the site via search or AI citation
- **Secondary:** Recruiters and engineering leaders quietly evaluating talent
- **Tertiary:** General tech enthusiasts, EV and Tesla community, camping/nature crossover

**Brand name:** `gekro` — always lowercase in UI, treated as a personal lab identity. Tagline suggestion: *"Building at the edge of what's possible."*

## PART 2: FULL TECH STACK (REASONED)

| Layer | Choice | Why |
|---|---|---|
| Framework | **Astro 4.x** | Zero JS by default, island architecture, perfect for content + interactive demos, blazing Lighthouse scores |
| Styling | **Tailwind CSS v4** | Utility-first, tree-shaken, pairs perfectly with Astro |
| Animations (Hero) | **Three.js + custom GLSL shaders** | WebGL particle field for the hero — mind-melting on first load |
| Animations (Page) | **GSAP + ScrollTrigger** | Industry standard for scroll-driven storytelling, smooth and GPU-accelerated |
| Animations (UI) | **Motion One** | Lightweight (~3kb), CSS-native, for micro-interactions and component reveals |
| CMS | **Sanity.io v3** | Headless, free tier, structured content, real-time, has a local Studio UI |
| Syntax Highlighting | **Shiki** | Renders to static HTML, zero client JS, gorgeous themes including custom ones |
| Slides / Presentations | **Reveal.js** wrapped in Astro components | Interactive web-native slides, embeddable, shareable via URL |
| Search | **Pagefind** | Runs at build time, fully static, zero server needed, works on Cloudflare Pages |
| Comments | **Giscus** | Powered by GitHub Discussions, zero cost, dev-credible, no ads |
| Email capture | **Beehiiv embed** | Free tier, beautiful, creator-native, clean API |
| Analytics | **Umami** (self-hosted on Pi) | Privacy-respecting, open source, no cookies, fits your green brand |
| Video embeds | Custom Lite-YouTube component | Deferred loading, no tracking cookies until user clicks |
| Monorepo | **Turborepo** | Lets blog site and future tools (packages) live in one repo with shared configs |
| Package manager | **pnpm** | Fast, efficient disk use, perfect for monorepo |
| Linting/Formatting | **ESLint + Prettier + Biome** | Enforced code quality — important since Antigravity writes code too |
| Testing | **Playwright** (E2E) + **Vitest** (unit) | Full coverage of critical paths |
| Deployment | **Cloudflare Pages** | Git-connected, global CDN, green infrastructure, preview URLs per branch |
| DNS | **Cloudflare DNS** (you already have this) | Zero config once Pages is connected |
| CI/CD | **GitHub Actions** | Runs lint, test, build checks before Cloudflare deploys |

## PART 5: ANIMATION STRATEGY (DETAILED)

### Hero — Mind-melting (Three.js / WebGL)
```
Effect: A 3D particle field representing a neural network / forest canopy hybrid.
- ~15,000 particles distributed in a sphere
- Particles drift slowly, connected by faint edges when close (force-directed graph feel)
- On mouse move: particles in cursor radius repel gently (magnetic field effect)
- On scroll: particles collapse inward (implosion) as user scrolls past hero
- Colour: forest green (#1a3a2a) base, electric teal (#00d4aa) highlights, cream (#f5f0e8) glow
- Performance: uses instanced mesh + buffer geometry for 60fps on mid-range hardware
- Fallback: CSS animated gradient for prefers-reduced-motion users
```

### Page Transitions — GSAP
```
- Route change: outgoing page fades + slides up (80px), incoming fades + slides from bottom
- Duration: 300ms — fast enough to feel snappy
- Implemented via Astro View Transitions API + GSAP override
```

### Scroll Storytelling — GSAP ScrollTrigger
```
- Homepage sections: each section pins briefly, content staggered in with 80ms delays
- Blog post: reading progress bar at top of viewport
- Experiments page: architecture diagrams draw themselves on scroll (SVG stroke animation)
- Topic pages: stat counters animate up when entering viewport
```

### Micro-interactions — Motion One
```
- All buttons: subtle scale(0.97) on press + background colour transition
- Cards: translateY(-4px) + box-shadow increase on hover
- Tag chips: background fill animation on hover
- Navigation links: underline draws from left on hover (CSS only)
- Code copy button: checkmark animation on success
```

### Performance Rules
```
- All Three.js code is lazy-loaded (dynamic import) — does not block page
- GSAP loads only on client, deferred
- prefers-reduced-motion: all animations disabled, static fallbacks shown
- Largest Contentful Paint target: < 1.8s on Cloudflare edge
```

## PART 9: n8n AUTOMATION WORKFLOW

### Architecture
```
New post published in Sanity
  → Sanity webhook (HTTP POST) → n8n (exposed via Cloudflare Tunnel on Pi)
    → n8n: parse post data (title, TLDR, URL, topics, aiSummary)
    → n8n: call Claude API (generate LinkedIn post from aiSummary + topics)
    → n8n: human approval gate (optional — sends you a Telegram/email for approval)
    → n8n: post to LinkedIn via LinkedIn node
    → n8n: (optional) post to X/Twitter via Twitter node
    → n8n: update Sanity post with "posted to socials" timestamp
```

### n8n Workflow: LinkedIn Post Generator Prompt
```
You are a ghostwriter for a technical AI engineer named [name] who goes by "gekro".
Write a LinkedIn post for the following new article.

Article title: {{title}}
Summary: {{aiSummary}}
Topics: {{topics}}
URL: {{url}}

Rules:
- 150-200 words max
- Start with a hook (a question or a surprising statement)
- 2-3 short paragraphs
- End with a clear CTA to read the full post
- Include 3-4 relevant hashtags at the end
- Tone: confident, builder-focused, no corporate speak
- Do NOT use the phrase "I'm excited to share"
```

### Cloudflare Tunnel Setup (for Pi)
```bash
# On your Pi, after Docker is running n8n:
cloudflared tunnel create gekro-n8n
cloudflared tunnel route dns gekro-n8n n8n.gekro.com
# Set DNS in Cloudflare dashboard: CNAME n8n → tunnel UUID.cfargotunnel.com
# This gives you https://n8n.gekro.com without exposing Pi's IP
```

## PART 10: DEPLOYMENT PIPELINE (GitHub → Cloudflare Pages)

### How the Bridge Works (Step by Step)
This is the piece you need held on to. Here's exactly what happens:

```
You write code → git push → GitHub → Cloudflare Pages (automatic)
```

**One-time setup:**
1. In Cloudflare dashboard → Pages → Create a project → Connect to Git
2. Authorise Cloudflare to access your GitHub account
3. Select your `gekro` repo
4. Build settings:
   - Framework preset: Astro
   - Build command: `pnpm run build --filter=web`
   - Build output directory: `apps/web/dist`
   - Root directory: `/` (repo root)
5. Add environment variables (Sanity project ID, dataset, API token)
6. Click Deploy

**After that, forever:**
- Push to `main` branch → Cloudflare builds and deploys to `gekro.com` automatically (takes ~60 seconds)
- Push to any other branch / open a PR → Cloudflare builds a **preview URL** like `abc123.gekro.pages.dev`
- You can share preview URLs to review before merging to main
- Antigravity makes a change → you push → preview URL appears → you review → merge → live

### Branch Strategy
```
main          → production (gekro.com)
dev           → your primary working branch
feature/*     → individual features (each gets its own preview URL)
content/*     → content-only branches (new posts, edits)
```

### GitHub Actions CI (`.github/workflows/ci.yml`)
```yaml
# Runs on every push to any branch
# Checks: pnpm install → lint → type-check → build
# If any step fails, Cloudflare will NOT deploy
# This protects production from broken pushes
```

### Local Development Flow
```bash
# In WSL terminal:
cd ~/projects/gekro
pnpm install                          # first time only
pnpm dev --filter=web                 # starts Astro dev server at localhost:4321
pnpm dev --filter=studio              # starts Sanity Studio at localhost:3333

# In a new tab — to write a new post:

                                      # creates draft in Sanity and opens Studio
```

```
### GEKRO.COM — ANTIGRAVITY WORKSPACE RULES v1.0

- Never hardcode hex values in components — always reference a CSS variable.

## File Naming Conventions
- Components: PascalCase.astro (e.g. PostCard.astro)
- Pages: lowercase kebab-case (e.g. about.astro, blog-archive.astro)
- Utilities: camelCase.ts (e.g. readingTime.ts)
- Sanity schemas: camelCase.ts (e.g. post.ts, slideDeck.ts)
- Never abbreviate — clarity over brevity in file names.

## Animation Rules
- Hero animations: Three.js only. Never animate the hero with CSS — it must be WebGL.
- All GSAP code must be wrapped in a check: `if (typeof window !== 'undefined')` to prevent SSR errors.
- All animations must include a `prefers-reduced-motion` media query fallback.
- Motion One is for UI micro-interactions only (buttons, cards, inputs).
- Scroll animations via GSAP ScrollTrigger only — do NOT use Intersection Observer manually.
- Animation durations: Hero = 1000-1500ms, Page transitions = 300ms, Micro = 150-200ms.

## Performance Rules
- Lighthouse score targets: Performance ≥ 95, Accessibility = 100, Best Practices = 100, SEO = 100.
- Three.js and GSAP must be dynamically imported (lazy-loaded), never in frontmatter.
- All third-party scripts (Giscus, Beehiiv, YouTube) must use `loading="lazy"` or be deferred.
- Images: use WebP format, always set width and height to prevent layout shift.
- No more than 2 web font families loaded. Use `font-display: swap`.

## SEO & AI Discoverability Rules
- Every page MUST include the SEOHead.astro component with populated title, description, and ogImage.
- Every blog post and experiment MUST include JSON-LD structured data via JsonLD.astro.
- Every post MUST have a populated `aiSummary` field in Sanity (2 sentences, plain text, no markdown).
- The TLDR.astro block must appear before the first H2 on every post.
- Canonical URLs must always be set explicitly.
- The public /api/posts.json endpoint must always stay live and return valid JSON.

## Content Generation Rules (when writing content for the blog)
- Write in first person as the site owner — confident, direct, no corporate vocabulary.
- Every post must include: a TLDR, at least one code block, and a conclusion with "what I learned."
- Technical accuracy first. Honesty about failures is preferred over polished success narratives.
- Never use: "In this article I will...", "In conclusion...", "It's worth noting that...", "I'm excited to share..."
- Code examples must be complete and runnable. No placeholder ellipsis `...` in code blocks.
- Always include estimated reading time (auto-computed — do not hardcode).

## Git & Deployment Rules
- Never commit directly to `main`. Always work on `dev` or a `feature/*` branch.
- Commit messages: conventional commits format — `feat:`, `fix:`, `content:`, `chore:`, `style:`
- Before pushing: run `pnpm lint` and `pnpm typecheck` from the repo root.
- Environment variables: NEVER hardcode API keys. Always use import.meta.env.
- The `.env.local` file is gitignored. Always update `.env.example` when adding new vars.

## Folder Rules
- New components go in the correct subfolder of src/components/ (blog/, ui/, hero/, etc.).
- New pages go in src/pages/. New layouts in src/layouts/. New utilities in src/lib/utils/.
- Never create files in the repo root. Shared config belongs in packages/.
- Sanity schema changes: always update both the schema file AND the TypeScript type definitions.

## Design System Rules
- Dark mode is the default and only mode (toggle may be added in Phase 2).
- Use the established colour palette CSS variables — no new colours without updating global.css first.
- Typography scale is defined in typography.css — do not override with arbitrary font sizes.
- Card components must use: surface background, border, subtle shadow, and hover state.
- Spacing: prefer Tailwind's scale (4px base). No arbitrary pixel values without comment.

## Accessibility Rules
```
